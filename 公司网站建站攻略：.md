# 公司网站建站攻略：

市面上确实有一些免费的网站空间可供选择，但它们在配置、使用限制和适用场景上各有不同。以下为您整理了几种不同类型的免费空间方案，供您参考：

**1. 国内传统虚拟主机（需备案）**

- **蓝队云**：提供永久免费的国内虚拟主机，配置为 1G 网页空间、300M MySQL 数据库，且不限流量和连接数。支持 PHP/ASP 环境，适合个人博客、简历站或开源项目演示。需要注意的是，由于是国内机房，必须进行实名认证和域名备案；此外，免费续期的条件通常要求在网站底部添加服务商的版权链接。

**2. 海外免备案空间**

- **彩虹云主机**：提供美国高防虚拟主机的免费版，拥有 500M 网页空间和 200M 共享 MySQL 数据库，采用美国 CN2 线路，无需备案即可使用。该空间适合用于个人网站体验、练习或非商业化建站，且自带域名防红功能，方便在社交平台推广。

**3. 静态网站托管平台（适合开发者）**
如果您搭建的是纯静态网站（如使用 Next.js、React、Vue、Hugo 等框架），可以考虑以下平台，它们都支持绑定自定义域名和免费 SSL：

- **Cloudflare Pages**：免费额度最大，提供无限带宽，国内访问速度相对较快，非常适合高流量网站或国内用户，但配置稍显复杂。
- **Vercel**：对 Next.js 等前端框架支持极佳，体验最好，但存在构建时长限制。
- **Netlify**：支持多种主流框架，自带表单功能，适合落地页和非技术用户，但有构建次数限制。
- **GitHub Pages**：简单稳定，适合个人博客和开源项目，但国内访问速度较慢。

**💡 避坑与选择建议：**

- **适用场景**：免费空间非常适合个人学习、博客、小型展示站或项目测试。如果是大型下载站、视频站或商业运营网站，建议购买付费云服务器。
- **数据安全**：虽然部分免费主机宣称稳定，但作为站长，务必养成定期手动备份数据库和网站文件的好习惯。
- **合规使用**：切勿利用免费空间搭建违法违规网站（如钓鱼、盗版等），正规服务商审核严格，一旦违规会直接封号。

### Cloudflare Pages 是 Cloudflare 推出的一项面向前端开发者的全栈平台服务，主要用于构建、部署和交付 Web 应用。你可以将其理解为类似 Vercel 或 Netlify 的静态站点托管服务，非常适合部署个人博客、文档站、前端工具以及部分 Next.js 应用。

以下是 Cloudflare Pages 的核心特点、适用场景及注意事项：

### 1. 核心优势与免费额度

Cloudflare Pages 的免费版对个人开发者非常友好，其最大的亮点在于**静态请求和带宽免费且不限量**。具体的免费额度包括：

- **价格**：永久免费（$0）。
- **站点与域名**：支持无限站点数量，每个项目支持绑定 100 个自定义域名。
- **构建限制**：每月最多 500 次构建，支持 1 个并发构建。
- **网络分发**：自带全球 CDN 极速分发、自动 HTTPS 证书以及免费 Web 分析功能。

### 2. 支持的项目类型

- **静态网站**：完美支持纯 HTML/CSS/JS、Vue、React、Astro、Hugo、VitePress、Docusaurus 等前端框架。
- **Next.js 应用**：支持静态导出的 Next.js 项目（输出到 `out` 目录）。对于包含 SSR、App Router、Server Actions 等特性的完整动态 Next.js 应用，官方推荐使用 Cloudflare Workers + OpenNext adapter 进行部署。

### 3. 部署方式

部署过程非常简便，主要支持以下两种方式：

- **连接 Git 仓库**：在控制台创建项目并绑定 GitHub 或 GitLab 仓库，代码提交后会自动触发构建和预览部署。
- **直接上传 / CLI 部署**：可以直接上传构建好的文件，或使用 Wrangler CLI 工具（如执行 `wrangler pages deploy ./dist`）进行部署。

### 4. 部署避坑指南

在部署纯 HTML 或静态项目时，有几个容易踩坑的细节需要注意：

- **构建命令配置**：对于不需要构建的纯 HTML 项目，必须将“Build command”留空或设置为 `exit 0`，并将“Output directory”设置为 `.`（英文句点）。如果不做此配置，系统可能会默认尝试执行构建命令而导致 404 错误。
- **文件路径与大小写**：`index.html` 必须放在仓库根目录，且文件名必须全小写（Pages 构建环境为 Linux，对大小写敏感）。此外，确保引用的图片、CSS 等路径与 Git 提交的树形结构完全一致。
- **域名与证书**：如果绑定了自定义域名，该域名的 DNS 必须由 Cloudflare 托管，否则自动生成的 HTTPS 证书将无法生效。

### 5. 适用场景推荐

- **高流量网站 / 国内用户**：得益于无限带宽和优秀的全球节点（包括亚洲节点），Cloudflare Pages 在国内的访问速度通常优于 GitHub Pages、Vercel 和 Netlify。
- **产品落地页 / 文档站**：零成本、高并发，非常适合用来做产品宣发或开源项目文档。
- **轻量级工具站**：如 JSON 格式化、Base64 编解码、时间戳转换等纯前端小工具。

总而言之，如果你的需求是托管静态站点或轻量级前端应用，Cloudflare Pages 凭借其“无限带宽”和“全球加速”的特性，是一个省心且高性价比的选择。但如果是复杂的后端重计算任务，则不建议强行使用。

### 部署 Cloudflare Pages 主要有三种常见方式，你可以根据自身的技术背景和项目需求选择最合适的一种：

### 1. 连接 Git 仓库（推荐，适合日常开发）

这是最主流且省心的部署方式，能够实现代码的自动化构建与部署。

- **操作方式**：在 Cloudflare Pages 控制台创建项目时，选择“连接现有 Git 仓库”，并授权关联你的 GitHub 或 GitLab 账号。
- **优势**：后续你只需在本地将代码 `push` 到仓库，Cloudflare 就会自动触发构建流程，并生成可供预览的部署版本，实现前后端一体化的高效开发。

### 2. 网页端直接上传（适合简单网页或新手）

如果你只有一个简单的静态网页，或者不想折腾 Git 操作，可以直接通过网页界面上传。

- **操作方式**：在控制台选择“直接上传（Drag and drop）”，将包含 `index.html`、CSS、JS 等文件的本地文件夹直接拖拽上传即可。
- **优势**：无需配置 Git，几分钟内就能获得一个公开的访问网址。

### 3. 使用 Wrangler CLI 部署（适合开发者）

对于习惯使用命令行的开发者，Cloudflare 提供了官方的 CLI 工具 Wrangler。

- **操作方式**：在终端中安装 Wrangler 后，运行 `wrangler pages deploy ./dist`（将 `./dist` 替换为你的实际构建产物目录）即可完成部署。

### 💡 核心避坑提醒（针对纯 HTML 项目）

无论你选择上述哪种方式，如果部署的是**不需要构建的纯 HTML 项目**，请务必注意以下配置，否则极易出现 404 错误：

- **构建命令（Build command）**：必须留空或设置为 `exit 0`。如果不做此配置，系统可能会默认尝试执行 `npm run build` 等命令，从而报错。
- **输出目录（Output directory）**：必须填写一个英文句点 `.`。这代表直接使用仓库根目录作为发布根目录，不能填 `./` 或留空。

### “连接 Git 仓库”是 Cloudflare Pages 最主流、最推荐的部署方式。它的核心优势在于**自动化**：一旦配置完成，你以后只需要把代码推送到 GitHub 或 GitLab，Cloudflare 就会自动帮你拉取代码、构建并发布到全球网络上，实现“一劳永逸”的更新。

以下是具体的操作步骤：

### 第一步：将代码上传至 GitHub

首先，你需要把本地的项目代码上传到 GitHub 仓库。

- **如果你熟悉命令行**：可以在终端中依次执行 `git init`、`git add .`、`git commit -m "Initial commit"`、`git branch -M main` 以及 `git push -u origin main` 等命令完成推送。
- **如果你是新手**：强烈建议下载 **GitHub Desktop** 图形化客户端。打开软件后，选择你的本地项目文件夹，点击提交（Commit），再点击发布仓库（Publish repository）即可。整个过程完全不需要敲任何 Git 命令。

### 第二步：在 Cloudflare 连接 GitHub

代码准备好后，就可以让 Cloudflare 接管了：

1. 登录 Cloudflare 控制台，进入“Workers & Pages”页面，点击“创建应用（Create application）”。
2. 选择“Pages”标签页，然后点击“连接现有 Git 仓库（Import an existing Git repository）”。
3. 首次使用会要求你进行授权。确认授权后，Cloudflare 会列出你 GitHub 账号下的仓库列表，找到你刚刚创建的项目，点击“开始设置（Begin setup）”。

### 第三步：配置构建信息（⚠️ 关键步骤）

在“设置构建和部署（Set up builds and deployments）”页面，你需要根据项目类型正确填写配置：

- **生产分支（Production branch）**：通常保持默认的 `main` 即可。

- 构建命令（Build command）

  ：

  - 如果是 **Vue、React 等需要构建的项目**，填写 `npm run build`。
  - 如果是 **纯 HTML/CSS/JS 项目**，**必须**填写 `exit 0` 或直接留空，否则系统会报错。

- 输出目录（Build output directory）

  ：

  - 需要构建的项目，通常填写 `dist`。
  - 纯 HTML 项目，**必须**填写一个英文句点 `.`（代表直接使用根目录），不能留空或填 `./`。

### 第四步：完成部署

配置无误后，点击“保存并部署（Save and Deploy）”。Cloudflare 会在几分钟内自动执行拉取代码、构建和发布流程。

部署成功后，页面会提示“Success”，并为你分配一个类似 `你的项目名.pages.dev` 的公开访问网址。

### 💡 后续如何更新网站？

采用这种方式后，你完全不需要再登录 Cloudflare 后台。以后每次在本地修改完代码，只需通过 GitHub Desktop 点击“Commit”并“Push”，或者在终端执行 `git push`，Cloudflare 就会自动感知到更新，并重新构建和发布你的网站。

- **解决冲突**：在拉取或合并代码遇到冲突时，GitHub Desktop 会提供可视化的冲突解决界面，支持逐行选择保留哪个版本的代码。

- **命令行协同**：对于变基（rebase）、cherry-pick 等复杂操作，命令行更具优势。您可以在 GitHub Desktop 的 `Repository` 菜单中勾选 `Show command output`，查看每个 GUI 操作对应的底层 Git 命令，边用边学。

### 要将你的 GitHub Pages 项目绑定到自定义域名，需完成**域名解析配置**和**GitHub Pages 设置**两步操作。以下是具体步骤：

### 一、在域名注册商处添加 DNS 记录

登录你购买域名的平台（如阿里云、腾讯云、GoDaddy、Namecheap 等），进入「DNS 管理」或「域名解析」页面，添加以下两条记录：

- A 记录（指向 GitHub Pages 服务器 IP）：
  - 主机记录：`@`（表示根域名，如 `example.com`）
  - 记录类型：`A`
  - 记录值：`185.199.108.153`（GitHub Pages 的 IPv4 地址之一，也可选 `185.199.109.153`、`185.199.110.153`、`185.199.111.153`，任选其一即可）
  - TTL：默认或 600 秒
- CNAME 记录（用于 www 子域名）：
  - 主机记录：`www`
  - 记录类型：`CNAME`
  - 记录值：`normal210.github.io`（替换为你的 GitHub Pages 用户名）
  - TTL：默认或 600 秒

> 注意：如果你只想用根域名（如 `example.com`），只需添加 A 记录；如果想同时支持 `www.example.com`，则需同时添加 A 记录和 CNAME 记录。

### 二、在 GitHub 仓库中配置自定义域名

1. 打开你的 GitHub 仓库（如 `sunrising-solar`）。
2. 点击顶部导航栏的 **Settings**。
3. 在左侧菜单中找到 **Pages**。
4. 在 “Custom domain” 输入框中填写你的完整域名（如 `example.com` 或 `www.example.com`）。
5. 点击 **Save**。
6. GitHub 会自动为你生成一个 `CNAME` 文件并推送到 `gh-pages` 分支，确保域名与仓库绑定。

### 三、等待 DNS 生效并验证

- DNS 记录添加后，通常需要 **几分钟到几小时** 才能全球生效（取决于 TTL 设置）。
- 你可以通过以下方式验证：
  - 在命令行运行 `ping example.com`，看是否返回 GitHub Pages 的 IP。
  - 直接在浏览器访问你的自定义域名，确认是否能正常显示网站内容。

### 四、可选：启用 HTTPS

GitHub Pages 默认为自定义域名提供免费的 HTTPS 证书，但需要满足以下条件：

- 域名已正确解析并绑定到 GitHub Pages。
- 在 GitHub Pages 设置中勾选 **Enforce HTTPS**（强制使用 HTTPS）。
- 等待几分钟，GitHub 会自动签发并部署 SSL 证书。

> 提示：如果你的域名是通过 Cloudflare 管理的，建议在 Cloudflare 中将代理状态设为“仅 DNS”（灰色云朵图标），避免影响 GitHub Pages 的证书签发。
